# container-tracking — Project Notes

## What this is
Automated ocean container tracking for Fantasia inbound POs. Pulls open container numbers from `fantasia-data.visibility`, syncs milestones from Shipsgo API, and writes results into this project's own Supabase tables (`containers`, `container_events`).

## Architecture
- **Own Supabase project:** `fantasia-tracking` (create under ed1fan org, $10/mo Pro add-on)
- **Reads from:** `fantasia-data` via `DATA_DB_URL` (data_reader role, read-only)
- **Writes to:** this project's `containers` and `container_events` tables
- **Shipsgo:** REST API, $2/container tracked, ~180 carriers

## Key source fields from fantasia-data.visibility
| Field | Meaning |
|---|---|
| `purchase_container_id` | Container number — Shipsgo tracking key |
| `purchase_supplier_reference` | BOL number |
| `purchase_import_number` | ERP inbound / confirmation number |
| `purchase_ship_via` | Carrier name (not sent to Shipsgo — auto-detected) |
| `purchase_port_name` | Destination port (POD) |
| `purchase_po_number` | PO number |
| `item_item_num_display` | Item identifier |
| `purchase_ship_orient_date` | **Internal ETD** |
| `time_date` | **Internal ETA** — primary ETA comparison field |

## ETA comparison
- **Internal ETA**: `visibility.time_date` (ERP expected arrival)
- **Shipsgo ETA**: `containers.eta` (carrier-reported)
- **Delta**: `containers.eta - visibility.time_date` (positive = late vs plan, negative = early)

## Exception logic
Evaluated after every sync. Each container can have multiple exceptions.

| Code | Severity | Condition |
|---|---|---|
| `OVERDUE` | 🔴 CRITICAL | Today > `containers.eta` AND status NOT IN (ARRIVED, DISCHARGED, UNTRACKED) |
| `DELAYED_VS_PLAN` | 🟠 WARNING | `containers.eta` > `visibility.time_date` by any amount |
| `ETA_SLIPPED` | 🟠 WARNING | Container has an `ETA_UPDATE` event in `container_events` (Shipsgo detected a shift from original ETA) |
| `ARRIVING_SOON` | 🟡 INFO | `containers.eta` within 7 days AND status NOT IN (ARRIVED, DISCHARGED) |

**Notes:**
- `UNTRACKED` status means Shipsgo lost tracking — flag it in the report but no ETA-based exception applies
- Containers with no Shipsgo ETA yet (INPROGRESS, NEW) are shown in the report but no exception is raised
- Multiple exceptions per container are all shown (e.g. a container can be both OVERDUE and DELAYED_VS_PLAN)

## Daily HTML report
- Generated by `sync/generate_report.py` after sync completes
- Dropped into `_reports/ContainerTracking_Report_YYYY-MM-DD.html`
- Power Automate watches `_reports/` and emails to `import@fantasia.com`
- Style matches photo tracker (Calibri, #1a3c5e header, inline CSS, email-safe)

**Report sections:**
1. Header — date + sync summary (synced count, errors, run time)
2. Exceptions table — grouped CRITICAL then WARNING then INFO (omitted if none)
3. All active containers table — sorted exceptions-first then by ETA ascending
   - Columns: Container ID | Status | Vessel | POD | Internal ETA | Shipsgo ETA | Delta | POs

**Row colors:**
- 🔴 OVERDUE: `#fde7e7`
- 🟠 DELAYED_VS_PLAN / ETA_SLIPPED: `#fff3e0`
- 🟡 ARRIVING_SOON: `#fffde7`
- ✅ ARRIVED / DISCHARGED: `#f0f7f0`
- Default: `#ffffff`

## Sync behavior
- Reads distinct `purchase_container_id` values from `fantasia-data.visibility` where not null
- Skips non-ISO container IDs (parcel numbers, N/A, TO FOLLOW)
- Registers new containers with Shipsgo (one-time, no carrier field — auto-detected)
- Fetches latest milestones for all registered containers
- Upserts `containers` table (ETA, status, vessel, etc.)
- Appends new events to `container_events` (deduped by container_id + event_type + event_date)
- Adds `import@fantasia.com` as follower on every container (409 = already added, OK)
- Rate limit: 0.7s delay between containers
